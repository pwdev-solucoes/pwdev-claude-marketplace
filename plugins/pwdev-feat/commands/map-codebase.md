# /pwdev-feat:map-codebase — Codebase Analysis

## Purpose

Analyze the current project and generate `.planning/feat/codebase.md` — a context document that the PWDEVIA planner uses to create better, more consistent action plans.

## Procedure

### STEP 1 — Stack Detection

```bash
# Package managers and deps
cat package.json 2>/dev/null | head -40
cat composer.json 2>/dev/null | head -40
cat requirements.txt 2>/dev/null | head -20
cat Gemfile 2>/dev/null | head -20
cat go.mod 2>/dev/null | head -10

# Framework detection
ls artisan 2>/dev/null && echo "LARAVEL"
ls nuxt.config.* 2>/dev/null && echo "NUXT"
ls next.config.* 2>/dev/null && echo "NEXT"
ls vite.config.* 2>/dev/null && echo "VITE"
```

### STEP 2 — Project Structure

```bash
# Directory structure (2 levels)
find . -maxdepth 2 -type d -not -path "*/node_modules/*" -not -path "*/.git/*" -not -path "*/vendor/*" -not -path "*/__pycache__/*" | sort | head -40

# Source file count by type
find . -type f \( -name "*.ts" -o -name "*.js" -o -name "*.vue" -o -name "*.php" -o -name "*.py" -o -name "*.jsx" -o -name "*.tsx" \) -not -path "*/node_modules/*" -not -path "*/vendor/*" | sed 's/.*\.//' | sort | uniq -c | sort -rn
```

### STEP 3 — Conventions

```bash
# Code style config
cat .editorconfig 2>/dev/null | head -20
cat .eslintrc* eslint.config.* 2>/dev/null | head -20
cat .prettierrc* 2>/dev/null | head -10
cat phpcs.xml* 2>/dev/null | head -10

# TypeScript config
cat tsconfig.json 2>/dev/null | head -20

# Git conventions
git log --oneline -10 2>/dev/null
```

### STEP 4 — Testing Setup

```bash
# Test framework detection
ls vitest.config.* jest.config.* phpunit.xml* pytest.ini 2>/dev/null
find tests/ test/ __tests__/ spec/ -type f 2>/dev/null | head -10
cat playwright.config.* 2>/dev/null | head -10
```

### STEP 5 — Generate codebase.md

Write to `.planning/feat/codebase.md`:

```markdown
# Codebase Context — {project name}

> Generated by /pwdev-feat:map-codebase on {date}
> Re-run after major structural changes

## Stack

| Technology | Version | Notes |
|------------|---------|-------|
| {lang/framework} | {version} | {notes} |

## Project Structure

```
{directory tree, 2 levels}
```

## Conventions

- **Code style:** {eslint/prettier/phpcs config}
- **Naming:** {PascalCase components, camelCase functions, etc.}
- **Imports:** {absolute/relative, aliases}
- **Commit style:** {conventional commits / other}
- **Architecture:** {MVC / Clean / DDD / other}

## Testing

- **Framework:** {vitest/jest/phpunit/pest/pytest}
- **E2E:** {playwright/cypress/none}
- **Test location:** {tests/ / __tests__ / etc.}
- **Coverage:** {configured / not configured}

## Key Files

| File | Purpose |
|------|---------|
| {path} | {what it does} |

## Patterns Observed

- {pattern 1: e.g., "Services return DTOs, not Eloquent models"}
- {pattern 2: e.g., "All forms use vee-validate + Zod"}
- {pattern 3}

## Notes

- {anything unusual or important for the planner}
```

### STEP 6 — Summary

```
✅ Codebase analyzed

📄 Context saved: .planning/feat/codebase.md

Stack: {summary}
Files: {N} source files
Tests: {framework} ({N} test files)
Patterns: {N} identified

👉 Next: /pwdev-feat:feat "description" to create your first plan
```

## Prohibitions
- NEVER read .env files (only .env.example)
- NEVER modify any project files — this is read-only analysis
- NEVER include secrets, tokens, or credentials in codebase.md
